=== INVESTIGATION FINDINGS ===

Root Cause:
- app/admin/monitoring/page.tsx reads NEXT_PUBLIC_ADMIN_API_SECRET (a server-only secret)
  from process.env and stores it in localStorage as "adminSecret"
- The secret is used as a Bearer token in Authorization headers to fetch metrics
- NEXT_PUBLIC_ prefix causes Next.js to embed ADMIN_API_SECRET value in the browser JS bundle
- Any XSS vulnerability (even self-contained) can read localStorage.getItem("adminSecret")
  or access window.__NEXT_DATA__ to steal the admin secret

Existing Auth Pattern:
- Supabase cookie-based session via createServerClient (@supabase/ssr)
- getServerSession() in lib/auth/get-server-session.ts
- useAuth() in app/providers/auth-provider.tsx tracks userRole ("admin"|"worker"|"business")
- Admin pages already check user session via useAuth()

Files in Scope:
- app/admin/monitoring/page.tsx (CLIENT - needs refactoring)
- app/api/admin/metrics/route.ts (SERVER - needs session auth)
- app/api/admin/cache-stats/route.ts (SERVER - needs session auth)
- .env.local.example (needs NEXT_PUBLIC_ADMIN_API_SECRET removed)
